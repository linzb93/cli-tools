# CLI 命令 / 功能调试器

这个 skill 用于在本地代码库中自动化调试和改 Bug 的流程。当用户描述某个 CLI 命令或功能的异常行为时，它会：

1. 从用户的问题中分析出涉及的命令 / 子命令 / 功能模块（例如 `token` 子命令）
2. 在项目根目录下的 `./packages/cli/src/bin.ts` 中搜索对应命令的定义和相关 import 语句
3. 将提取到的命令代码复制到 `./packages/cli/src/bin-test.ts` 中，并清理其中之前其他命令的代码
4. 保留 bin-test.ts 中既有的基础结构（如 program 初始化、hook 设置等），只在指定注释区域内替换命令实现
5. 运行 `npm run dev` 启动调试开发模式，后续通过 `myclit` 作为入口命令进行调试

在调试模式下，会优先通过 `myclit` 命令来复现和验证问题，例如：

-   `myclit token xxx`：用来解析某个 token 的内容，帮助检查解析逻辑是否正确

## 使用方法

当用户说：

-   "帮我调试一下 xxx 命令"
-   "这个 xxx 功能有 Bug，帮我修一下"
-   "执行 myclit xxx 的时候报错了，帮我看看"
-   "解析这个 token 有问题，帮我排查一下"

其中 xxx 是命令名称或功能名称（如 token, ai, git, beauty 等）。

## 处理流程

1. 解析用户请求，提取出关键命令名称、参数和期望行为
2. 在 `bin.ts` 中定位对应命令实现，将其同步到 `bin-test.ts` 指定区域
3. 根据用户的问题在相应命令或业务模块中修改代码（修复 Bug 或调整逻辑）
4. 执行 `npm run dev` 启动调试模式
5. 在调试模式下，根据问题类型构造合适的调试命令，例如：
    - 使用 `myclit token xxx` 解析用户提供的 token
    - 使用其它 `myclit ...` 子命令或项目内已有的调试 / 测试命令
6. 观察命令输出，将结果作为调试依据返回给用户，必要时继续修改代码并重复上述步骤

## 注意事项

-   从 `bin.ts` 同步代码到 `bin-test.ts` 时，只替换 `//**** 请在这里替换需要调试的代码 ****` 注释之间的内容
-   确保正确处理 import 语句依赖关系，避免在 bin-test.ts 中出现未引入的符号
-   保持 bin-test.ts 的基础结构不变（program 初始化、hook 设置等）
-   修改命令实现或业务逻辑时代码变更范围要尽量小，避免影响无关功能
-   在运行调试命令前，确保当前工作目录在项目根目录，并通过 `npm run dev` 启动调试模式后再使用 `myclit`
-   当问题与 token 解析相关时，应优先通过 `myclit token xxx` 来复现和验证
-   如需长时间调试，可保持调试终端持续运行，按需多次执行 `myclit ...` 命令
