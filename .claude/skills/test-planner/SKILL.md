# 测试用例规划专家

## 功能概述

本 Skill 专注于为 CLI 命令或业务模块规划全面的测试策略。它通过分析设计文档 (`docs/implement.md`) 或源代码，识别核心业务逻辑、边界条件和异常处理场景，生成测试计划供用户确认，并最终编写高质量的 Vitest 测试用例。在必要时，它还会建议或执行代码重构以提高可测试性。

## 使用场景

- 当用户询问 "某某命令有哪些功能可以测试？"
- 当用户要求 "为这个功能规划测试用例"
- 当用户需要从零开始为某个模块编写测试，且希望先确认测试范围时
- 当现有代码结构难以测试，需要重构建议时

## 工作流程

### 1. 智能分析阶段

系统会自动定位目标功能的业务代码目录，并按以下优先级获取上下文：
1.  **优先读取**：`docs/implement.md`（设计文档），理解功能预期行为和业务规则。
2.  **降级读取**：`index.ts` 或其他业务源代码，通过静态分析提取逻辑分支。

### 2. 测试规划阶段

基于分析结果，系统会以 Markdown 列表形式输出 **测试计划草案**，包含：
-   ✅ **Happy Path（正常流程）**：核心业务功能验证。
-   ⚠️ **Edge Cases（边界条件）**：空值、极大极小值、特殊字符等。
-   ❌ **Error Handling（错误处理）**：网络异常、权限不足、配置错误等。
-   🔧 **Refactoring Needs（重构建议）**：如果发现代码难以测试（如逻辑耦合、私有方法），会建议拆分函数或导出模块。

**示例输出：**
```markdown
## 测试计划：Git 分支删除功能

### 核心功能
- [ ] 删除本地分支（无未合并更改）
- [ ] 强制删除本地分支（含未合并更改）
- [ ] 删除远程分支

### 边界条件
- [ ] 尝试删除当前所在分支（应阻止）
- [ ] 分支名不存在
- [ ] 批量删除多个分支

### 错误处理
- [ ] Git 命令执行失败（如网络断开）
- [ ] 用户取消确认操作
```

### 3. 用户确认与重构

- 等待用户确认测试计划。
- 如果涉及代码重构（如 `export` 私有函数、拆分大函数），在此阶段执行代码修改。

### 4. 测试生成阶段

根据确认的计划，生成 `index.test.ts` 文件。

#### 代码实现规范（参考 test-generator）

-   **文件位置**：同级目录下的 `__tests__/index.test.ts`
-   **测试框架**：Vitest
-   **设计模式**：AAA (Arrange, Act, Assert)
-   **原则**：
    -   **原子化**：每个测试只验证一个行为。
    -   **Mocking**：对于外部依赖（如 Git 操作、文件系统、API 调用），使用 `vi.mock` 进行模拟。
    -   **环境控制**：使用 `process.env.VITEST` 隔离测试环境特有逻辑。

#### 代码模板

```typescript
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { deleteBranch } from '../index';

vi.mock('node:child_process'); // 示例：Mock 外部依赖

describe('deleteBranch', () => {
  beforeEach(() => {
    vi.clearAllMocks();
  });

  it('应该成功删除存在的本地分支', async () => {
    // Arrange
    const branchName = 'feature/test';
    
    // Act
    await deleteBranch(branchName);

    // Assert
    expect(...)
  });
});
```

## 质量保障

-   **文档驱动**：优先基于设计文档，确保测试符合业务初衷。
-   **可测试性优先**：敢于重构代码以适应测试，而不是为了测试编写复杂的 Hack 代码。
-   **覆盖率**：确保覆盖所有规划的场景。
