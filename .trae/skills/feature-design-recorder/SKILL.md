# 功能设计记录器

这个 skill 帮助您在 `docs/designs/` 目录下记录计划开发的功能，并通过「问答 → 方案对比 → 设计文档」的方式，把想法整理成可实施的设计。

## 使用场景

当用户说"我想开发 xx 功能"或"我想开发一个新命令 xxx"、"我想为 yyy 命令添加一个功能"时：

1. 识别是「新命令」还是「为已有命令加功能」
2. 通过多轮问答澄清需求与使用场景
3. 基于需求生成一个或多个可选实现方案
4. 列出每个方案大致需要修改的代码位置（只记录，不修改）
5. 在用户选择最终方案后，根据复杂度和用户意愿决定是否生成设计文档

## 对话流程

### 1. 触发与意图识别

-   从用户输入中识别：
    -   是否为新命令
    -   是否为某个现有命令新增功能
    -   命令名称（如：git、eng 等）

### 2. 需求澄清的多轮问答

在生成任何设计文档之前，必须先通过多轮问答把需求问清楚，典型问题包括但不限于：

-   目标与场景：
    -   这个命令/功能主要解决什么问题？
    -   典型使用场景是什么？
-   调用方式：
    -   希望用户如何调用？是新命令、子命令，还是增加参数/选项？
    -   有哪些必须参数和可选参数？是否有默认值？
-   输入与输出：
    -   输入来自哪里（命令行参数、配置文件、标准输入等）？
    -   期望输出形式是什么（纯文本、表格、JSON、文件等）？
-   约束与边界：
    -   有无性能/时延方面的要求？
    -   是否需要兼容现有行为？遇到错误时希望如何反馈？

在收集完信息后，用中文对需求做一次总结，向用户回读并确认，确认后才进入方案生成阶段。

### 3. 生成可选实现方案

在需求确认后，至少生成 1 个可选方案；如果存在不同实现思路，可以给出多个方案。每个方案需要包含：

-   接口设计
    -   命令/子命令/参数的形式
    -   与现有命令体系的一致性
-   行为说明
    -   成功时的行为和输出
    -   错误和异常情况下的行为
-   影响范围与修改位置（设计层面）
    -   仅列出可能需要修改或新增的模块/文件，不做任何修改
    -   例如：
        -   命令注册或路由文件
        -   业务逻辑模块
        -   类型定义或公共工具模块
-   优缺点
    -   可维护性
    -   对现有用户的影响
    -   实现复杂度等

### 4. 用户选择方案并最终确认

-   引导用户从上述方案中选择一个，或提出调整意见
-   根据用户的选择和调整，再次整理出「最终实现方案」的描述
-   用户确认后，进入是否生成设计文档的判断

### 5. 是否生成设计文档

在最终方案确认后，由用户决定本次是否需要生成设计文档：

-   适合生成设计文档的场景（默认建议生成）：
    -   新命令开发
    -   涉及多个模块或较大范围改动
    -   未来自己可能会频繁修改、容易忘记细节的功能
-   可以不生成设计文档的场景：
    -   只是在现有命令上增加/调整一个简单参数
    -   只是改动输出文案或格式等非常简单的修改

交互规则：

-   Skill 应明确询问用户：“本次功能是否需要生成设计文档？简单改动可以只在对话中约定，不创建文件。”
-   如果用户选择生成设计文档：
    -   按「文件内容结构」章节的模板，生成完整的中文设计文档
    -   文档中包含需求总结、方案对比、最终实现方案、设计层面的「修改点一览」
    -   在 `docs/designs/` 目录下创建 markdown 文件
-   如果用户选择不生成设计文档：
    -   仅在当前对话中给出最终方案的清晰中文总结
    -   不在 `docs/designs/` 目录下创建任何文件

无论是否生成设计文档，本 skill 都不会修改任何代码文件。

### 6. 开发确认

在设计文档生成完成（或最终方案确认）后，**必须**明确询问用户：

> "是否需要立即按照此设计文档进行代码开发？"

-   **如果用户同意**：才可以进入开发阶段，开始修改代码。
-   **如果用户不同意或未明确指令**：任务结束，不进行任何代码修改。

**注意**：严禁在生成文档后不经询问直接开始写代码。

## 核心原则

**⚠️ 设计阶段禁止修改代码**

-   在方案设计和文档生成阶段，绝不修改任何源代码文件
-   仅在用户在「开发确认」环节明确同意后，方可进行代码开发
-   只创建文档文件，不触碰任何 .ts、.js、.json 等代码文件（除非进入开发阶段）
-   如果用户在设计阶段要求修改代码，必须明确拒绝并说明先完成设计

## 语言要求

**重要：全程使用中文生成文档**

-   所有功能描述和文档内容必须使用中文
-   文件名中的英文描述除外（保持技术命名规范）
-   plan 模式讨论和最终生成的.md 文档都必须是中文
-   代码分析和建议也要用中文表达

## 文件命名规范

-   使用当前日期格式：YYYY-MM-DD
-   添加功能的简短英文描述
-   用连字符分隔单词
-   保持简洁且具有描述性

示例：

-   `20240218-user-authentication.md`
-   `20240218-api-rate-limiting.md`
-   `20240218-dashboard-analytics.md`

## 业务代码位置说明

如果是为某个命令添加功能，业务代码位于：`/Users/linzhibin/Documents/coding/cli-tools/packages/cli/src/business` 目录下

**命令与目录对应关系：**

-   `eng` 命令 → `translate` 目录
-   其他命令 → 同名目录（如 `git` 命令对应 `git` 目录）

**分析步骤：**

1.  识别用户提到的命令名称
2.  根据对应关系找到正确的业务目录
3.  分析现有代码结构和服务文件
4.  在功能记录中注明相关代码位置
